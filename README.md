# Telegram-бот для отслеживания акций MOEX

Личный Telegram-бот проверяет российские акции из `tickers.txt` через MOEX ISS API и ищет тикеры по условию:

```text
close последней закрытой свечи > high предыдущей закрытой свечи
```

Простыми словами: акция закрылась выше максимума предыдущей свечи.
Процент в строке `Пробой high` считается одинаково для ручных и автоматических отчётов: `(close_last - high_previous) / high_previous * 100`.

Бот использует только закрытые свечи в timezone `Europe/Moscow`. Текущая незакрытая свеча не используется. Поле MOEX `end` может меняться внутри формирующейся свечи, поэтому закрытие определяется по `begin` и границе таймфрейма:

- `1m`: `begin + 1 минута`;
- `10m`: `begin + 10 минут`;
- `1h`: `begin + 1 час`;
- `1d`: начало следующего календарного дня;
- `1w`: начало следующей календарной недели;
- `1mo`: начало следующего календарного месяца.

Например, свеча `16:10–16:19` становится закрытой только в `16:20 МСК`.

## Таймфреймы

Доступны ручная проверка и автоуведомления для всех таймфреймов:

- `1m` — 1 минута, MOEX ISS `interval=1`;
- `10m` — 10 минут, MOEX ISS `interval=10`;
- `1h` — 1 час, MOEX ISS `interval=60`;
- `1d` — 1 день, MOEX ISS `interval=24`;
- `1w` — 1 неделя, MOEX ISS `interval=7`;
- `1mo` — 1 месяц, MOEX ISS `interval=31`.

Свечи `1m` и `10m` берутся напрямую из MOEX ISS. Ручной агрегации коротких свечей в коде нет.

## Ручная проверка

Кнопка `🔍 Проверить сейчас` работает для всех таймфреймов: `1m`, `10m`, `1h`, `1d`, `1w`, `1mo`.

Бот:

1. получает свечи из MOEX ISS;
2. сортирует их по `begin/end`;
3. отбрасывает текущую формирующуюся свечу и выбирает две последние полностью закрытые свечи;
4. сравнивает `close_last > high_previous`;
5. показывает тикеры, которые прошли условие;
6. добавляет период последней и предыдущей свечи;
7. показывает оборот в рублях, если MOEX вернул `value`.

Если последняя доступная закрытая свеча отстаёт от ожидаемой по текущему времени, ручная проверка всё равно показывает последний доступный отчёт, но добавляет предупреждение с ожидаемым и фактически доступным периодом. Например, для `1h` в `19:01 МСК` ожидается свеча `18:00–18:59 МСК`; если MOEX пока отдаёт только `17:00–17:59 МСК`, бот покажет предупреждение перед отчётом.

Статусы `X2`, `X3`, `X4` в ручной проверке не показываются.

## Автоуведомления

Автоуведомления работают сразу по нескольким таймфреймам. По умолчанию включены базовые уведомления:

- `1d` — 1 день;
- `1w` — 1 неделя;
- `1mo` — 1 месяц.

В разделе `🔔 Уведомления` пользователь может дополнительно включить короткие таймфреймы `1m`, `10m` и `1h`. Например, если включить `10m`, бот будет отправлять уведомления по закрытию 10-минутной свечи, а базовые уведомления за 1 день, 1 неделю и 1 месяц останутся включёнными.

Для `10m` бот проверяет появление новой закрытой 10-минутной свечи и отправляет отчёт, если найдёт тикеры, у которых:

```text
close последней закрытой 10-минутной свечи > high предыдущей закрытой 10-минутной свечи
```

Если дополнительно включён `1h`, бот в минуты `01–20` каждого часа сначала вычисляет ожидаемый `candle_key` последней закрытой часовой свечи. Например, в `19:01 МСК` ожидается ключ `18:00`. Если MOEX пока отдаёт только `17:00`, автоотчёт не отправляется, `last_processed_candle_keys` и `last_sent_candle_keys` не обновляются, а в лог пишется причина. В следующую минуту бот пробует снова и отправляет отчёт только после появления ожидаемой свечи.

При `SEND_EMPTY_REPORTS_FOR_HIGHER_TIMEFRAMES=true` для `1h`, `1d`, `1w` и `1mo` отправляется отчёт даже при отсутствии подходящих тикеров. Для `1m` и `10m` пустые отчёты не отправляются. При значении `false` пустые отчёты не отправляются ни для одного таймфрейма.

Защита от дублей использует отдельные обработанные и отправленные ключи для каждого таймфрейма: `notification_timeframe + candle_begin`. Примеры ключей: `2026-06-04 16:10` для `10m`, `2026-W23` для `1w`, `2026-06` для `1mo`.

APScheduler запускает короткий триггер примерно раз в минуту, но запросы к MOEX выполняются только для включённых таймфреймов, у которых могла появиться новая закрытая свеча:

- `1m` — каждый запуск планировщика;
- `10m` — только после очередной 10-минутной границы и задержки `INTRADAY_CHECK_DELAY_SECONDS`;
- `1h` — каждую минуту в диапазоне `01–HOURLY_CHECK_WINDOW_MINUTES` после начала нового часа;
- `1d` — в `DAILY_REPORT_TIME`;
- `1w` — в день `WEEKLY_REPORT_DAY` и время `WEEKLY_REPORT_TIME`;
- `1mo` — один раз в день в `MONTHLY_REPORT_TIME` только в конце месяца и первые дни нового месяца.

Если таймфрейм сейчас проверять рано, он пропускается без запросов к MOEX. Таймфреймы, которые нужно проверить в одном запуске, обрабатываются параллельно через `asyncio.to_thread`.

Долгий проход автоуведомлений выполняется отдельной фоновой задачей. APScheduler запускает только короткий триггер и не создаёт второй проход, пока предыдущий не завершён. Дополнительно `run_auto_notifications` защищён `asyncio.Lock`. Поэтому синхронные запросы MOEX не блокируют обработку Telegram-кнопок.

В `bot.log` записываются включённые, проверяемые и пропущенные таймфреймы, причина пропуска, длительность всего запуска, длительность каждого таймфрейма и количество HTTP-запросов к MOEX. Для `1h` дополнительно записываются ожидаемый `candle_key`, последний доступный `candle_key`, последние пять свечей каждого тикера, их `begin/end/close/high`, признак закрытия, выбранные последние две закрытые свечи, итог `sent/skip` и причина пропуска.

## Статусы X2/X3/X4/...

Статусы `X2`, `X3`, `X4` и далее без ограничения показываются только в автоуведомлениях.

- `X2` означает, что тикер второй раз подряд попал в выборку на разных закрытых свечах выбранного таймфрейма.
- `X3` означает, что тикер третий раз подряд попал в выборку на разных закрытых свечах.
- Повторная обработка одного `candle_key` не увеличивает счётчик.
- Если тикер перестал попадать в очередную закрытую свечу или бот пропустил промежуточную свечу, его счётчик по этому таймфрейму сбрасывается.

Счётчики считаются отдельно для каждого таймфрейма. Например, `SBER` может иметь `X3` на `10m` и не иметь статуса на `1h`.

## Оборот

В отчётах не показывается `volume`. Вместо объёма бот показывает оборот в рублях из поля MOEX `value`, если это поле пришло в ответе API.

Примеры формата оборота:

- `523000` → `523 тыс ₽`;
- `5520000` → `5.52 млн ₽`;
- `5520000000` → `5.52 млрд ₽`.

Если MOEX не вернул `value`, бот не падает, не показывает строку оборота и пишет предупреждение в `bot.log`.

## Настройки пользователя

В `user_settings.json` для пользователя хранятся:

- `chat_id`;
- `user_id`;
- `selected_timeframe` — таймфрейм ручной проверки;
- `notification_timeframes` — список таймфреймов автоуведомлений;
- `auto_notifications_enabled` — включены ли уведомления;
- `selected_tickers` — персональный список тикеров, которые пользователь отслеживает;
- `last_processed_candle_keys` — последние успешно обработанные ключи закрытых свечей отдельно по таймфреймам;
- `last_sent_candle_keys` — последние ключи свечей, по которым реально отправлено Telegram-сообщение;
- `streaks` — счётчики повторного попадания тикеров по таймфреймам.

Пример:

```json
{
  "selected_timeframe": "1h",
  "notification_timeframes": ["1m", "10m", "1h", "1d", "1w", "1mo"],
  "auto_notifications_enabled": true,
  "selected_tickers": ["SBER", "GAZP", "LKOH", "ROSN"],
  "last_processed_candle_keys": {
    "1m": "2026-06-04 14:10",
    "10m": "2026-06-04 16:10",
    "1h": "2026-06-04 14:00"
  },
  "last_sent_candle_keys": {
    "1m": "2026-06-04 14:10",
    "10m": "2026-06-04 16:10",
    "1h": "2026-06-04 14:00"
  },
  "streaks": {
    "10m": {
      "SBER": 5,
      "NVTK": 2
    },
    "1h": {
      "LKOH": 4
    }
  }
}
```

Старое поле `last_sent_candle_times` автоматически мигрирует в `last_sent_candle_keys` при загрузке настроек. Существующие отправленные ключи также копируются в `last_processed_candle_keys`, чтобы после обновления не появились дубли.

Кнопка `🔄 Рестарт` не перезапускает программу и не удаляет историю Telegram. Она сбрасывает настройки пользователя к значениям:

```text
selected_timeframe = 1d
notification_timeframes = 1d, 1w, 1mo
auto_notifications_enabled = true
selected_tickers = все тикеры из доступного списка
last_processed_candle_keys = пусто
last_sent_candle_keys = пусто
streaks = пусто
```

## Создание Telegram-бота

1. Откройте Telegram и найдите `@BotFather`.
2. Отправьте `/newbot`.
3. Введите имя бота.
4. Введите username, который заканчивается на `bot`.
5. Скопируйте токен в `.env` как `TELEGRAM_BOT_TOKEN`.

Не публикуйте `.env` и не отправляйте токен посторонним.

## Настройка `.env`

Создайте `.env` из примера:

```powershell
copy .env.example .env
notepad .env
```

Пример:

```dotenv
TELEGRAM_BOT_TOKEN=ваш_токен_от_BotFather
TELEGRAM_CHAT_ID=
ALLOWED_USER_ID=
TIMEZONE=Europe/Moscow
MARKET=MOEX
DEFAULT_TIMEFRAME=1d
DEFAULT_NOTIFICATION_TIMEFRAMES=1d,1w,1mo
AUTO_NOTIFICATIONS=true
USER_SETTINGS_PATH=user_settings.json

TICKERS_FILE=tickers.txt
LOG_FILE=bot.log

MOEX_BOARD=TQBR
MOEX_REQUEST_TIMEOUT=20
MOEX_REQUEST_RETRIES=3
AUTO_SCHEDULER_INTERVAL_SECONDS=60
INTRADAY_CHECK_DELAY_SECONDS=30
HOURLY_CHECK_WINDOW_MINUTES=20

DAILY_REPORT_TIME=23:55
WEEKLY_REPORT_DAY=FRIDAY
WEEKLY_REPORT_TIME=23:55
MONTHLY_REPORT_TIME=23:55
SEND_EMPTY_REPORTS_FOR_HIGHER_TIMEFRAMES=true
```

`TIMEZONE` должен оставаться `Europe/Moscow`, потому что границы закрытия свечей и ключи уведомлений считаются по московскому времени.

Старые переменные `MOEX_TIMEOUT_SECONDS` и `SCHEDULER_INTERVAL_SECONDS` поддерживаются как резервные, но для новых настроек используйте `MOEX_REQUEST_TIMEOUT` и `AUTO_SCHEDULER_INTERVAL_SECONDS`.

Настройка `SEND_EMPTY_REPORTS_FOR_HIGHER_TIMEFRAMES` управляет только пустыми автоотчётами `1h/1d/1w/1mo`. Для `1m/10m` пустые сообщения не отправляются независимо от этой настройки.

Настройка `HOURLY_CHECK_WINDOW_MINUTES` задаёт, сколько минут после начала нового часа бот будет повторно проверять появление ожидаемой закрытой свечи `1h`. Значение по умолчанию — `20`.

MOEX-клиент переиспользует одну `requests.Session` в пределах проверки таймфрейма и повторяет временно неудачные запросы до `MOEX_REQUEST_RETRIES` раз. Для свечей клиент после чтения cursor сразу переходит к последней нужной странице, не загружая все промежуточные страницы диапазона. Полные ошибки пишутся в `bot.log`; в Telegram отображается только краткий список временно недоступных тикеров.

`TELEGRAM_CHAT_ID` можно оставить пустым. После `/start` бот сохранит `chat_id` в `user_settings.json`.

Чтобы ограничить доступ одним пользователем, укажите:

```dotenv
ALLOWED_USER_ID=123456789
```

Старые переменные отдельных daily/weekly/monthly-уведомлений больше не нужны. Новая логика использует список `notification_timeframes`. Базовые `1d`, `1w` и `1mo` добавляются автоматически.

## Тикеры

Заполните `tickers.txt`, по одному тикеру на строку:

```text
SBER
GAZP
LKOH
YDEX
```

Пустые строки и комментарии после `#` игнорируются.

## Выбор тикеров

У каждого пользователя есть персональный список `selected_tickers`. При первом запуске и при рестарте бот выбирает все доступные тикеры по умолчанию.

Кнопка `📋 Мои тикеры` открывает экран выбора. На одной странице показывается до 20 тикеров, поэтому длинные списки разбиваются на страницы. Тикер с `✅` включён в отслеживание, тикер с `❌` исключён. Нажатие на тикер сразу переключает состояние и сохраняет настройки.

Доступные действия:

- `✅ Выбрать все` — включает все доступные тикеры;
- `Снять все` — очищает список и показывает предупреждение;
- `Сохранить` — оставляет пользователя на экране выбора и показывает количество выбранных тикеров;
- `Главное меню` — возвращает в главное меню.

Ручная проверка, отчёт по обороту и автоуведомления используют только `selected_tickers` текущего пользователя. Если список пустой, ручная проверка покажет сообщение: `У вас не выбрано ни одного тикера. Откройте Мои тикеры и выберите акции для отслеживания.` Автоуведомления для такого пользователя пропускаются без Telegram-спама и пишут причину в лог.

Если `tickers.txt` пустой, бот пытается получить список российских акций через Tinkoff Invest API. Для этого нужен токен в `TINKOFF_INVEST_TOKEN` или `TINKOFF_TOKEN` и установленный пакет `tinkoff-investments`.

## Кнопки

- `▶️ Старт` — текущие настройки и источник данных.
- `🔄 Рестарт` — сброс настроек без удаления истории Telegram.
- `🔍 Проверить сейчас` — ручной отчёт по выбранному таймфрейму.
- `⏱ Таймфрейм` — выбор таймфрейма ручной проверки.
- `🔔 Уведомления` — базовые `1d/1w/1mo`, выбор дополнительных коротких таймфреймов и общий переключатель.
- `📄 Последний отчёт` — последний сохранённый ручной отчёт.
- `💰 Оборот` — топ-10 тикеров по обороту последней закрытой свечи.
- `📋 Мои тикеры` — персональный выбор тикеров для ручной проверки и автоуведомлений.
- `⚙️ Настройки` — таймфреймы, статус уведомлений, источник данных, timezone и логика отбора.
- `❓ Помощь` — краткая справка.

Основные inline callback-data:

```text
⬅️ Главное меню = main_menu
🔄 Обновить / 🔍 Проверить сейчас = refresh
⏱ Таймфрейм = timeframe_menu
```

Каждый callback сразу подтверждается через `query.answer()` и записывается в `bot.log`. Старые callback-data `menu:main`, `report:check` и `menu:timeframe` поддерживаются для кнопок в ранее отправленных сообщениях.

## Запуск на Windows

Откройте PowerShell в папке проекта:

```powershell
cd C:\Users\admin\Desktop\test-codex
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
python -m pip install -r requirements.txt
python -m unittest discover -s tests -v
python bot.py
```

Если PowerShell запрещает активацию:

```powershell
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
.\.venv\Scripts\Activate.ps1
```
